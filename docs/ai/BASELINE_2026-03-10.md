## 6. Следующий этап: operational hardening

Следующий этап больше не про срочные fix-ы, а про аккуратное hardening поверх стабильного baseline.

### 1. Re-audit before any new server-side apply
- Все live facts в baseline относятся к состоянию, проверенному на дату `2026-03-10`.
- Перед любым новым server-side apply сначала делать свежий narrow read-only audit по нужному contour.

### 2. Optional housekeeping only
- Рассматривать только по отдельной задаче:
  - cleanup stale `S1` path for `okdesk-pipeline`
  - prompt/memory compatibility cleanup only if a real loader need appears
  - future architectural migration of `okdesk-pipeline` only if requirements change

### 3. Model routing hardening only as separate project
- Если когда-либо понадобится simplification:
  - сначала architecture decision
  - потом отдельный rollback-ready server plan
- До этого layered split считать accepted live reality

### 4. Baseline maintenance
- Держать канон, backlog и changelog в sync с live audits.
- Snapshot docs не переписывать как live truth.
- Для новых drift-пунктов сначала audit, потом docs, потом apply only if needed.
